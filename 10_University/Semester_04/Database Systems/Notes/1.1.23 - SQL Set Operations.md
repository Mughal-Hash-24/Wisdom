# Technical breakdown: SQL Set Operations

## 1. Surgical Definition (Internals)
SQL Set Operations (UNION, INTERSECT, EXCEPT/MINUS) are the practical implementation of **Relational Algebra** set theory. Unlike JOINs, which combine columns horizontally based on a predicate, set operations combine result sets **vertically**.

From a compiler's perspective, these are not just "merging lists." The query optimizer treats set operations as **multi-source inputs** that must conform to a strict structural contract: identical column counts and compatible data types. For operations like `UNION` (which implies distinctness), the engine inserts a hidden `DISTINCT` operator, typically implemented via a **Hash Aggregate** or a **Sort-Unique** operation, to prune duplicates across the combined buffers.

## 2. The Laboratory (Proof of Concept)
To demonstrate the engine's behavior, we compare a standard `UNION` with its high-performance counterpart, `UNION ALL`.

```sql
-- Create two sets of IDs
WITH SetA AS (SELECT 1 AS id UNION ALL SELECT 2),
     SetB AS (SELECT 2 AS id UNION ALL SELECT 3)

-- 1. UNION: Combines and removes duplicates (Costly Sort/Hash)
SELECT id FROM SetA
UNION
SELECT id FROM SetB; -- Output: 1, 2, 3

-- 2. INTERSECT: Finds the overlap (Logic: A ∩ B)
SELECT id FROM SetA
INTERSECT
SELECT id FROM SetB; -- Output: 2

-- 3. EXCEPT (MINUS in Oracle): Difference (Logic: A - B)
SELECT id FROM SetA
EXCEPT
SELECT id FROM SetB; -- Output: 1
```

## 3. Memory & System Context
* **Stack/Heap:** SQL engines use **TempDB** or **Work Tables** in memory (or disk if the set exceeds the buffer pool) to perform the deduplication required by `UNION`, `INTERSECT`, and `EXCEPT`.
* **Cost:** `UNION ALL` is the "zero-cost" operator because it simply concatenates stream pointers. `UNION` introduces $O(N \log N)$ complexity due to the required sorting or $O(N)$ for hashing to ensure uniqueness.
* **Life-cycle:** The execution plan usually involves a "Concatenation" operator followed by a "Sort" or "Hash Match" depending on the cardinality estimates.

## 4. Best Practices & Anti-Patterns
* **The "Union All" Default:** Always prefer `UNION ALL` unless business logic strictly requires unique rows. It avoids the massive CPU/IO overhead of deduplication.
* **Column Alignment:** Ensure data types match exactly (e.g., `VARCHAR` vs `NVARCHAR` can trigger implicit conversions that break index usage).
* **Predicate Pushing:** Set operations often act as an optimization barrier. Try to apply `WHERE` filters *inside* each subquery rather than on the final combined result set to reduce memory pressure.

# Technical breakdown: SQL Set Operations

## 1. Surgical Definition (Internals)
Set operations (UNION, INTERSECT, EXCEPT) are algebraic operators that combine the results of two or more independent queries into a single result set. Unlike JOINs, which expand relations horizontally by correlating columns based on a predicate, set operations combine relations vertically.

**Engine View:** The SQL query optimizer treats set operators as "Physical Operators" that manipulate data streams. 
- **UNION ALL:** The most efficient; it is a simple stream concatenation (Append).
- **UNION / INTERSECT / EXCEPT:** These are "Blocking Operators." The engine cannot stream results immediately because it must first identify and remove duplicates. To do this, it typically employs a **Hash Aggregate** (building a hash table in memory to track seen rows) or a **Sort-Unique** operation (sorting the input and discarding consecutive identical rows).

## 2. The Laboratory (Proof of Concept)
The following script demonstrates the behavior of the three primary set operations using two mock tables representing different team assignments.

```sql
-- Initializing the environment
CREATE TABLE Team_A (emp_id INT, emp_name VARCHAR(50));
CREATE TABLE Team_B (emp_id INT, emp_name VARCHAR(50));

INSERT INTO Team_A VALUES (1, 'Alice'), (2, 'Bob');
INSERT INTO Team_B VALUES (2, 'Bob'), (3, 'Charlie');

-- 1. UNION (Combines and removes duplicates)
SELECT * FROM Team_A UNION SELECT * FROM Team_B; 
-- Output: (1, Alice), (2, Bob), (3, Charlie)

-- 2. INTERSECT (Finds the overlap)
SELECT * FROM Team_A INTERSECT SELECT * FROM Team_B; 
-- Output: (2, Bob)

-- 3. EXCEPT (Subtracts the second set from the first)
SELECT * FROM Team_A EXCEPT SELECT * FROM Team_B; 
-- Output: (1, Alice)
```

## 3. Memory & System Context
* **Stack/Heap:** Intermediate result sets are materialized in the database's **Temporary Workspace** (e.g., TempDB in SQL Server, Work Mem in PostgreSQL).
* **Cost:** 
    - **UNION ALL:** $O(N + M)$ - Near zero overhead.
    - **UNION:** $O(N \log N)$ or $O(N)$ depending on if it uses Sort or Hash. It is significantly more expensive than UNION ALL.
* **Life-cycle:** The memory allocated for the hash table or sort buffer is released once the operator completes its scan. However, for massive datasets, this can trigger "Spilling to Disk," where the operation is offloaded to the physical drive, causing a massive performance bottleneck.

## 4. Best Practices & Anti-Patterns
* **The "UNION ALL" Rule:** Always prefer `UNION ALL` unless you explicitly require deduplication. Using `UNION` unnecessarily is a common performance drain.
* **Type Alignment:** Ensure columns in the same position have compatible data types. Mismatched types force the engine to perform "Implicit Conversion," which prevents index usage.
* **Anti-Pattern:** Using set operations to replace complex JOIN logic. While readable, set operations can sometimes be less efficient than a well-tuned SEMI-JOIN or ANTI-JOIN because they force the materialization of the entire result set before comparison.

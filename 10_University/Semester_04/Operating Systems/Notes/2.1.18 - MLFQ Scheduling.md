## Architecture of the Multilevel Feedback Queue
A Multilevel Feedback Queue (MLFQ) is a dynamic scheduling framework that organizes the ready state into a hierarchy of discrete queues ($Q_0, Q_1, ..., Q_n$), each characterized by a specific priority level and scheduling policy (typically Round Robin). Unlike static priority scheduling, MLFQ utilizes a process's execution historyÃ¢â‚¬â€specifically its CPU burst durationÃ¢â‚¬â€as a feedback signal to adjust its priority level. The system initializes new processes at the highest priority queue ($Q_0$). As a process consumes its allocated time quantum without performing an I/O operation, the scheduler interprets this as a transition toward a CPU-bound state and demotes the process to the next lower-priority queue ($Q_{i+1}$), which typically features a longer time quantum to accommodate larger computational loads while minimizing context-switch overhead.

## The Feedback Mechanism and Priority Evolution
The core logic of MLFQ is governed by the principle of penalizing CPU-intensive tasks to preserve low response times for interactive, I/O-bound tasks. The state transition is triggered by the nature of the process's completion within its quantum:

1.  **Priority Maintenance:** If a process yields the CPU before its quantum expires (e.g., waiting for I/O), it retains its current priority level or is promoted. This ensures that interactive tasks (short bursts) remain in high-priority queues.
2.  **Priority Demotion:** If a process exhausts its entire time quantum, it is preempted and moved to the tail of the next lower-priority queue.
3.  **Starvation Prevention (Aging):** To prevent low-priority, long-running processes from being starved by a constant stream of high-priority arrivals, the system employs "Priority Boosting." Periodically, all processes are migrated back to the highest-priority queue, resetting their "history" and ensuring they receive some CPU time.

## Performance Dynamics and Parameters
The behavior of an MLFQ scheduler is defined by five critical parameters: the number of queues, the scheduling algorithm for each queue (e.g., RR or FCFS), the criteria for promoting a process, the criteria for demoting a process, and the initial queue assignment.

| Queue | Priority | Quantum | Strategy |
| :--- | :--- | :--- | :--- |
| $Q_0$ | Highest | 8ms | Round Robin |
| $Q_1$ | Medium | 16ms | Round Robin |
| $Q_2$ | Lowest | - | FCFS |

Mathematically, MLFQ attempts to minimize the Total Wait Time and Response Time without prior knowledge of the process burst lengths, essentially approximating a Shortest Job First (SJF) scheduler through empirical observation. Its primary failure mode occurs when the "Boosting" interval is too large, leading to starvation, or when a process "games" the scheduler by performing a trivial I/O operation just before its quantum expires to stay in the high-priority queue (often referred to as "gaming the scheduler").

## Architecture and Core Philosophy of MLFQ
> **Seed:** "@expand Explain the fundamental architecture of the Multilevel Feedback Queue (MLFQ) scheduling algorithm. Focus on the core philosophy of dynamic priority adjustment and how the 'feedback' mechanism distinguishes it from static Multilevel Queue scheduling."

## The Adaptive Architecture of MLFQ
Multilevel Feedback Queue (MLFQ) is a dynamic scheduling discipline that manages processes using a hierarchy of distinct queues, each representing a different priority level. Unlike static schedulers that require a priori knowledge of a process's execution time, MLFQ uses the execution history of a process as a heuristic to predict its future behavior. Its primary objective is to minimize response time for interactive (I/O-bound) jobs while simultaneously ensuring that long-running (CPU-bound) jobs eventually complete.

The system is structured as a set of $n$ queues: $Q_0, Q_1, \dots, Q_{n-1}$. 
- **$Q_0$ (Highest Priority):** Assigned the smallest time quantum (e.g., 8ms).
- **$Q_i$ (Intermediate):** Assigned increasing time quanta as priority decreases.
- **$Q_{n-1}$ (Lowest Priority):** Often scheduled via First-Come-First-Served (FCFS) or a very large Round Robin (RR) quantum.

## The Feedback Mechanism: Dynamic Priority Adjustment
The "Feedback" in MLFQ refers to the systemÃ¢â‚¬â„¢s ability to observe a process's behavior and adjust its position in the queue hierarchy accordingly. This distinguishes it from static Multilevel Queue (MLQ) scheduling, where a process is permanently assigned to a specific queue (e.g., "System," "Interactive," or "Batch") based on its type at creation.

### The Rules of Movement
MLFQ operates on a set of fundamental rules (originally formalized by Fernando CorbatÃƒÂ³):
1. **Rule 1:** If $Priority(A) > Priority(B)$, A runs (B does not).
2. **Rule 2:** If $Priority(A) = Priority(B)$, A and B run in RR using the quantum of the current queue.
3. **Rule 3:** When a job enters the system, it is placed at the highest priority ($Q_0$).
4. **Rule 4 (The Feedback/Demotion Rule):** 
    - If a job uses its entire time slice while running, its priority is reduced (it is moved down one queue).
    - If a job gives up the CPU (e.g., for I/O) before its time slice expires, it stays at the same priority level.

### The Behavioral Heuristic
By demoting jobs that consume their full time slice, MLFQ identifies "CPU hogs." These processes are relegated to lower-priority queues with longer time slices, where they won't interfere with the responsiveness of the system. Conversely, interactive processes that perform frequent I/O naturally stay at higher priorities because they rarely exhaust their time quanta, ensuring they are serviced almost immediately when they become ready.

## Static MLQ vs. Dynamic MLFQ
The core differentiator is **flexibility and overhead**. 

| Feature | Static Multilevel Queue (MLQ) | Multilevel Feedback Queue (MLFQ) |
| :--- | :--- | :--- |
| **Assignment** | Fixed at process creation. | Dynamic based on CPU/IO behavior. |
| **Adaptability** | None. A batch job remains low priority forever. | High. A process can change "nature" and move. |
| **Knowledge** | Requires process classification (System/User). | Requires no prior knowledge; learns by doing. |
| **Feedback Loop**| None. Queues are isolated silos. | Active. The "feedback" of quantum exhaustion triggers migration. |

## Failure Modes: Starvation and Gaming the System
MLFQ is susceptible to two primary failure modes:
1. **Starvation:** If the system is saturated with high-frequency, short-duration interactive jobs, processes in the lower-priority queues ($Q_{n-1}$) will never receive CPU time.
2. **Gaming the Scheduler:** A savvy programmer could write a process that performs a meaningless I/O operation just before its time slice expires (e.g., writing to a null device). This would trick the "feedback" mechanism into keeping the process at the highest priority level, effectively stealing CPU time from others.

### The Solution: Priority Boosting
To solve starvation and reset "gamers," a **Priority Boost** mechanism is implemented. After some period $S$, the system moves all jobs in the system to the topmost queue ($Q_0$). This ensures that even the most CPU-bound background task will occasionally get a window of high-priority execution and prevents any process from being permanently buried in the lower tiers.

## The Operational Rules of Process Movement
> **Seed:** "@expand Detail the specific rules that govern how processes move between queues in an MLFQ. Explain the logic for initial queue placement, the conditions for priority demotion (time-slice expiration), and how the algorithm handles processes that voluntarily relinquish the CPU."

The Multi-Level Feedback Queue (MLFQ) is a dynamic scheduling heuristic that manages process priority based on observed execution behavior. It aims to minimize response time for interactive jobs (short bursts) while preventing starvation and managing throughput for CPU-intensive jobs (long bursts). It operates on the principle of "learn from the past to predict the future," where the scheduler assumes every process might be short until proven otherwise.

## The Five Fundamental Rules of MLFQ
The movement of processes is governed by a set of rigid rules that define the lifecycle of a task within the priority hierarchy:

1. **Rule 1:** If Priority(A) > Priority(B), A runs (B doesn't).
2. **Rule 2:** If Priority(A) = Priority(B), A & B run in Round Robin (RR) using the time slice of the given queue.
3. **Rule 3 (Admission):** When a job enters the system, it is placed at the highest priority (the topmost queue).
4. **Rule 4 (Demotion):** Once a job uses up its time allotment at a given level (regardless of how many times it has given up the CPU), its priority is reduced (i.e., it moves down one queue).
5. **Rule 5 (Priority Boost):** After some time period $S$, move all the jobs in the system to the topmost queue to prevent starvation.

## Initial Placement and Optimistic Scheduling
MLFQ implements an "optimistic" entry policy. Because the scheduler has no *a priori* knowledge of a process's length or burst behavior, it grants every new process the benefit of the doubt by placing it in the highest priority queue ($Q_{max}$). 

This ensures that short, interactive tasks (e.g., a keystroke handler or a mouse click event) are serviced immediately. If the job is indeed short, it will complete and exit the system before its first time slice expires, achieving near-optimal response time without ever being demoted.

## Priority Demotion: The Penalty for CPU-Intensity
Demotion is triggered by the exhaustion of a queue's **Time Allotment**. In basic implementations, this was simply the time slice. In modern, robust MLFQ implementations, it is the *cumulative* time spent at that level.

- **Mechanism:** The scheduler maintains a `time_consumed` counter for each process at its current level.
- **Trigger:** If `time_consumed >= budget_at_level_K`, the process is moved to $Q_{K-1}$.
- **Result:** CPU-bound processes (like a heavy mathematical simulation) naturally "sink" to the bottom queues. These lower queues typically have longer time slices to reduce context-switching overhead for long-running tasks, whereas higher queues have short slices for rapid responsiveness.

## Voluntary Relinquishment and the Anti-Gaming Problem
A critical design challenge in MLFQ is handling processes that yield the CPU before their time slice expiresÃ¢â‚¬â€usually to perform I/O.

### The Naive Approach (The "Gaming" Vulnerability)
In older MLFQ versions, Rule 4 stated: "If a process yields the CPU before its slice is up, its priority stays the same." This created a massive security hole: a process could "game" the scheduler by performing a trivial I/O operation (like a write to the console) at 99% of its time slice. By yielding just before the limit, it would reset its timer and stay in the high-priority queue indefinitely, effectively monopolizing the CPU and starving other tasks.

### The Robust Solution: Better Accounting
To prevent gaming, modern MLFQ uses **Cumulative Time Accounting**. Instead of resetting the process's time slice upon an I/O yield, the scheduler remembers how much time the process has used *at that priority level*. 

- **Logic:** If a process uses 8ms of a 10ms slice and then yields for I/O, when it returns, it only has 2ms left at that priority. 
- **Internal State:**
  ```c
  struct process_control_block {
      int priority_level;
      float total_time_at_current_level;
      float time_limit_for_level;
  };

  void on_timer_interrupt(Process P) {
      P.total_time_at_current_level += clock_tick;
      if (P.total_time_at_current_level >= P.time_limit_for_level) {
          demote_process(P);
          P.total_time_at_current_level = 0;
      }
  }

  void on_io_yield(Process P) {
      // Prioritize the process for return, but do NOT reset total_time_at_current_level
      move_to_wait_queue(P);
  }
  ```

## The Priority Boost: Handling State Changes
A failure mode of pure MLFQ is that a process's behavior might change. A CPU-bound process that "sank" to the bottom might become interactive later. Without a mechanism to move back up, it would be trapped in the low-priority slow lane forever. The **Priority Boost** (Rule 5) periodically flushes the system, moving everyone to the top. This solves two problems:
1. **Starvation:** Bottom-tier processes are guaranteed to run eventually.
2. **Dynamic Behavior:** Processes that have become interactive can re-establish their high-priority status by staying at the top after the boost.

## MLFQ as an Approximation of Shortest Job First (SJF)

## The Predictive Sieve: Approximating SJF
Multi-Level Feedback Queue (MLFQ) is an adaptive scheduling mechanism designed to minimize turnaround timeÃ¢â‚¬â€the primary metric optimized by Shortest Job First (SJF)Ã¢â‚¬â€without the prerequisite knowledge of a process's next CPU burst length ($t_n$). While SJF is theoretically optimal, it requires an oracle to predict the future. MLFQ replaces this oracle with an inductive inference engine: it assumes that a processÃ¢â‚¬â„¢s immediate past behavior is the best predictor of its immediate future.

The system functions as a series of hierarchical filters. New processes are optimistically granted the highest priority ($Q_{max}$), assuming they are short. If a process exhausts its allotted time slice (quantum) without yielding the CPU, it is demoted to a lower priority queue. This demotion is a mathematical signal that the process is CPU-bound rather than I/O-bound.

## Mathematical Basis: The Power Law of Bursts
The logical foundation of MLFQ rests on the empirical observation of process behavior distributions. In most computing environments, CPU bursts follow an exponential or power-law distribution: the majority of bursts are very short, while a small minority are extremely long.

If $T_{burst}$ is the total time a process needs to run before blocked by I/O or termination:
- **Hypothesis:** If a process $P$ has consumed $k$ units of time and has not finished, the probability $P(T_{burst} > k + \Delta | T_{burst} > k)$ remains high for CPU-bound tasks.
- **SJF Objective:** $\min \sum_{i=1}^{n} F_i$, where $F$ is the finish time.
- **MLFQ Approximation:** By prioritizing processes that haven't used much CPU time yet, MLFQ ensures that short jobs (the majority) exit the system rapidly, reducing the average waiting time exactly as SJF does.

## The Learning Mechanism: Priority as an Estimator
The scheduler "learns" by adjusting the priority level $L$ based on execution history. The priority level serves as an implicit estimator of the process's remaining burst time.

### State Transition Logic
The transition can be modeled by the following logic:
1. **Rule 1:** If Priority($A$) > Priority($B$), $A$ runs ($B$ waits).
2. **Rule 2:** If Priority($A$) = Priority($B$), $A$ & $B$ run in Round Robin (RR).
3. **Rule 3:** When a job enters the system, it is placed at the highest priority ($Q_0$).
4. **Rule 4a:** If a job uses up an entire time slice while running, its priority is reduced (it moves down one queue).
5. **Rule 4b:** If a job gives up the CPU before the time slice is up (e.g., for I/O), it stays at the same priority level.

```python

# Simplified MLFQ Priority Logic
def update_priority(process, quantum_used, total_quantum):
    if quantum_used == total_quantum:
        # Process is likely a long-running batch job
        process.priority = max(0, process.priority - 1)
    elif quantum_used < total_quantum:
        # Process yielded for I/O; likely interactive or short
        # Maintain high priority to ensure low response time
        process.priority = process.priority 
```

By penalizing jobs that consume full quanta, MLFQ effectively pushes long-running "background" tasks to the bottom, while allowing "foreground" interactive tasks or short-lived scripts to stay at the top. This mimics SJF because the short jobs naturally stay in higher queues and finish before the long jobs in lower queues ever get significant CPU time.

## Internal Dynamics: The Time-Slice Stretch
To maintain efficiency, MLFQ implementations typically vary the time quantum across queues.
- **$Q_0$ (Highest):** Small quantum (e.g., 10ms). High responsiveness for very short jobs.
- **$Q_1$:** Medium quantum (e.g., 40ms).
- **$Q_n$ (Lowest):** Large quantum (e.g., 100ms) or FCFS. Efficient throughput for heavy batch jobs.

This structure ensures that if a job is long, it isn't context-switched frequently once it has been "identified" as a long job, reducing overhead while the system focuses its switching energy on the uncertain/short jobs at the top.

## Failure Modes and Correction (Priority Boost)
The primary failure mode of a pure MLFQ is **starvation**. If the system is constantly flooded with short-lived tasks, jobs in the lower queues ($Q_n$) will never execute.

To correct this, systems implement a **Priority Boost** (Rule 5): 
- *After some time period $S$, move all the jobs in the system to the topmost queue.*

Mathematically, this resets the "learned" history, allowing the scheduler to re-evaluate processes that might have changed from CPU-bound to I/O-bound, and ensuring that even the longest batch jobs make progress (guaranteeing liveness). Without the boost, the "learning" is permanent and unforgiving, which breaks the dynamic nature required for modern multi-tasking.

## Addressing Starvation and Gaming: The Priority Boost

A Multi-Level Feedback Queue (MLFQ) scheduler attempts to optimize turnaround time for short jobs while maintaining responsiveness for interactive jobs by using execution history to predict future behavior. However, the basic MLFQ implementationÃ¢â‚¬â€which lowers a process's priority every time it exhausts its time sliceÃ¢â‚¬â€introduces critical failure modes where processes can either be indefinitely denied CPU cycles (starvation) or maliciously manipulate the scheduler to maintain high-priority status (gaming).

## MLFQ Vulnerabilities: Starvation and Exploitation

The fundamental logic of MLFQ relies on the assumption that if a process uses its full time slice, it is likely a CPU-bound batch job and should be demoted to a lower-priority queue. This heuristic creates two primary vulnerabilities:

### 1. The Starvation Problem
In a system with a constant influx of interactive or short-lived jobs, these processes will saturate the high-priority queues. Because MLFQ strictly prioritizes higher queues over lower ones, a long-running CPU-bound process relegated to the bottom queue may never receive a CPU cycle. The schedulerÃ¢â‚¬â„¢s strict hierarchy transforms a "low priority" into a "zero progress" state if the system's "interactive load" exceeds the total CPU capacity.

### 2. Gaming the Scheduler
A malicious or poorly written process can "game" the scheduler by artificially yielding the CPU just before its time slice expires. In a basic MLFQ, priority is often maintained if a process performs an I/O operation before the quantum ends. 
- **Mechanism:** A process with a 10ms quantum runs for 9.9ms and then issues a trivial I/O (e.g., a read to a dummy file). 
- **Result:** The scheduler sees the process yielded voluntarily, assumes it is "interactive," and keeps it in the highest priority queue. This allows a CPU-bound process to hog nearly 100% of the CPU while masquerading as a high-priority interactive task.

## The Priority Boost Mechanism

To resolve these vulnerabilities, a global reset mechanism known as the **Priority Boost** is implemented as Rule 5 in the standard MLFQ model.

### Technical Implementation
The system defines a constant period $S$ (the boost interval). After every $S$ milliseconds, the scheduler performs a global state transition:
1. All processes currently in the system, regardless of their current queue level or execution history, are moved to the top-most (highest priority) queue.
2. All accumulated "accounting" data regarding time slice usage for the current priority level is reset.

```text
// Pseudocode for Priority Boost
if (system_clock - last_boost_time >= S) {
    for each process p in all_queues {
        move_to_highest_queue(p);
        reset_time_slice_accounting(p);
    }
    last_boost_time = system_clock;
}
```

### Analogy: The Corporate Tenure Reset
Think of MLFQ as a corporate ladder where employees who finish projects slowly are demoted. Eventually, the basement is full of hard workers who are ignored because "new hires" (interactive jobs) keep getting the executive suite's attention. The **Priority Boost** is a mandatory company-wide reorganization every quarter where everyone is promoted back to the top floor. Those who are truly productive (interactive) will stay near the top, while those who are slow (CPU-bound) will eventually drift back down, but everyone is guaranteed at least a few days of executive-level attention every quarter.

## Liveness and Fairness Guarantees

The Priority Boost provides two mathematical guarantees for the operating system's liveness:

1. **Starvation Prevention:** By moving every job to the top queue every $S$ ms, the scheduler guarantees that every process will receive a share of the CPU at least once every $S$ interval. The "liveness" of a process is no longer dependent on the behavior of other processes, but on the global constant $S$.
2. **Neutralizing Exploits:** Gaming the scheduler relies on maintaining a specific state in the priority hierarchy. The Priority Boost forcibly destroys this state. Even if a process successfully games the system to stay at the top, it eventually shares that space with the CPU-bound processes it was trying to starve, ensuring that no single process can permanently monopolize the highest tier through I/O manipulation.

### The $S$ Parameter Trade-off
The effectiveness of the mechanism depends entirely on the value of $S$. 
- **If $S$ is too large:** Processes may still starve for unacceptable durations.
- **If $S$ is too small:** The system reverts to a Round Robin-like behavior, and the benefits of MLFQ (prioritizing interactive jobs) are lost to the frequent global resets. 

In production kernels, $S$ is typically tuned to ensure that the "cost" of the boost is amortized over the gains in fairness.

## The Architecture: A Multi-Tiered Filtration System

A Multilevel Feedback Queue (MLFQ) acts like a series of increasingly coarse filters in a factory: light, fast-moving items (interactive jobs) are cleared quickly at the top, while heavy, slow-moving blocks (CPU-bound jobs) sink to the bottom where they are handled in bulk.

In this 3-queue system, the tiers are defined as follows:

| Queue | Priority | Scheduling Policy | Time Quantum | Purpose |
| :--- | :--- | :--- | :--- | :--- |
| **Q0** | Highest | Round Robin (RR) | 8 ms | Short bursts, I/O-bound responsiveness. |
| **Q1** | Medium | Round Robin (RR) | 16 ms | Longer CPU bursts, transitioning jobs. |
| **Q2** | Lowest | First-Come, First-Served (FCFS) | N/A | Background, batch, or "CPU-hog" jobs. |

### The Core Rules
1. **Rule of Arrival:** All new jobs enter **Q0**.
2. **Rule of Demotion:** If a job uses its entire time quantum in its current queue without yielding (blocking for I/O), it is moved to the next lower priority queue (Q0 Ã¢â€ â€™ Q1 Ã¢â€ â€™ Q2).
3. **Rule of Retention:** If a job yields the CPU before its quantum expires (e.g., to perform I/O), it stays at the same priority level.
4. **Rule of Priority:** A job in a lower-priority queue can only run if all higher-priority queues are empty.

## Complexity Analysis

### Time Complexity: $O(1)$ (Scheduling)
The scheduler only needs to look at the head of the highest-priority non-empty queue to find the next job. This is a constant-time operation. However, "Priority Boosting" (moving all jobs to Q0 periodically to prevent starvation) is an $O(N)$ operation where $N$ is the number of processes.

### Space Complexity: $O(N)$
Each process requires a Process Control Block (PCB) and an entry in one of the queue data structures. The memory overhead scales linearly with the number of active processes.

## Implementation (C++ Pseudocode)

```cpp
struct Process {
    int id;
    int remaining_time;
    int current_queue; // 0, 1, or 2
};

void mlfq_schedule(vector<queue<Process*>>& levels) {
    while (true) {
        Process* current = nullptr;
        int level_idx = -1;

        // Find the highest priority non-empty queue
        for (int i = 0; i < 3; ++i) {
            if (!levels[i].empty()) {
                current = levels[i].front();
                levels[i].pop();
                level_idx = i;
                break;
            }
        }

        if (!current) break; // All queues empty

        int quantum = (level_idx == 0) ? 8 : (level_idx == 1) ? 16 : current->remaining_time;
        int used = execute(current, quantum); // Returns time used or -1 if I/O

        if (used == -1) {
            // Job blocked for I/O; stays at current level
            add_to_io_wait(current); 
        } else if (used >= quantum && level_idx < 2) {
            // Job used full quantum; demote
            current->current_queue++;
            levels[current->current_queue].push(current);
        } else if (current->remaining_time > 0) {
            // Job partially used quantum but didn't block; stays in current level
            levels[level_idx].push(current);
        }
    }
}
```

## Edge Cases and Potential Bugs

1.  **Starvation:**
    *   **Behavior:** A constant stream of short interactive jobs (Q0) prevents long-running CPU jobs in Q2 from ever receiving service.
    *   **Solution:** Implement "Priority Boosting." Every $S$ milliseconds, move all processes to Q0 regardless of their history.
    *   **Common Bug:** Setting the boost interval $S$ too high, leading to noticeable "lag" for background tasks that occasionally need to be responsive.

2.  **Gaming the System:**
    *   **Behavior:** A malicious or poorly written program executes for 99% of its quantum and then performs a trivial I/O operation to keep its high-priority status in Q0.
    *   **Solution:** Use "Time Accounting." Track the total time a process spends at a level. Once the cumulative time exceeds the quantum, demote it, even if it yielded multiple times.
    *   **Common Bug:** Reselling the full quantum on every I/O return without checking total residency time.

3.  **Empty High-Priority Queues:**
    *   **Behavior:** If Q0 and Q1 are empty, the scheduler must default to Q2 (FCFS).
    *   **Expected Behavior:** A new arrival in Q0 must immediately preempt a running Q2 job to maintain the system's responsiveness guarantee.
    *   **Common Bug:** Failing to check for higher-priority arrivals during the execution of a long FCFS burst in Q2.

- **Q2**: First-Come, First-Served (FCFS) (Lowest Priority)

### 2. The Gantt Chart

0    2      4      6        10     12     14       18     20        24

### 3. Step-by-Step Logic
- **T=0**: **P1** arrives and enters **Q0**. It runs for the $TQ=2$.
- **T=1**: **P2** arrives. It waits in **Q0** as P1 is currently occupying the CPU.
- **T=2**: **P1** exhausts its $TQ$. It still has 6ms left. It is demoted to **Q1**. Simultaneously, **P3** arrives and enters **Q0**. 
  - *Status*: Q0: [P2, P3], Q1: [P1].
- **T=2-4**: **P2** (top of Q0) runs for $TQ=2$. It has 2ms remaining and is demoted to **Q1**.
  - *Status*: Q0: [P3], Q1: [P1, P2].
- **T=4-6**: **P3** (top of Q0) runs for $TQ=2$. It has 8ms remaining and is demoted to **Q1**.
  - *Status*: Q0: [], Q1: [P1, P2, P3].
- **T=6-10**: Q0 is empty, so we move to Q1. **P1** runs for $TQ=4$. It has 2ms remaining and is demoted to **Q2**.
- **T=10**: **P4** arrives and enters **Q0**. Since Q0 has higher priority, it preempts/starts before other queues.
  - *Status*: Q0: [P4], Q1: [P2, P3], Q2: [P1].
- **T=10-12**: **P4** runs for $TQ=2$. It finishes completely.
- **T=12-14**: Q0 is empty. **P2** (top of Q1) runs. It only needs 2ms to finish. **P2 completes.**
- **T=14-18**: **P3** (top of Q1) runs for $TQ=4$. It has 4ms remaining and is demoted to **Q2**.
- **T=18-20**: Q0 and Q1 are empty. **P1** (top of Q2/FCFS) runs. It needs 2ms and finishes.
- **T=20-24**: **P3** (last in Q2) runs for its remaining 4ms and finishes.

## Complexity Analysis

| Metric | Complexity | Why? |
| :--- | :--- | :--- |
| **Time** | $O(N \cdot K)$ | Where $N$ is the number of processes and $K$ is the number of priority levels. Each process is inserted and removed from a queue a constant number of times ($K$ at most). |
| **Space** | $O(N)$ | The scheduler must maintain queues to store the process control blocks (PCBs) or identifiers for all $N$ processes. |

## Python Implementation (MLFQ Simulation)

```python
from collections import deque

class Process:
    def __init__(self, name, arrival, burst):
        self.name = name
        self.arrival = arrival
        self.burst = burst
        self.rem = burst

def mlfq_schedule(processes, quanta):
    time = 0
    queues = [deque() for _ in range(len(quanta) + 1)] # Q0, Q1... + FCFS
    completed = []
    
    # Sort processes by arrival
    processes.sort(key=lambda x: x.arrival)
    proc_idx = 0
    
    while len(completed) < len(processes):
        # 1. Add arriving processes to Q0
        while proc_idx < len(processes) and processes[proc_idx].arrival <= time:
            queues[0].append(processes[proc_idx])
            proc_idx += 1
            
        # 2. Find the highest priority non-empty queue
        target_q = -1
        for i in range(len(queues)):
            if queues[i]:
                target_q = i
                break
        
        if target_q == -1:
            time += 1 # Idle
            continue
            
        p = queues[target_q].popleft()
        
        # 3. Execute
        # If last queue, use FCFS (effectively infinite quantum)
        q_val = quanta[target_q] if target_q < len(quanta) else p.rem
        run_time = min(p.rem, q_val)
        
        # In a real OS, we'd check for new arrivals during run_time (preemption)
        # Here we simulate the jump
        p.rem -= run_time
        time += run_time
        
        # 4. Feedback Logic
        if p.rem > 0:
            # Demote if not already in the bottom queue
            next_q = min(target_q + 1, len(queues) - 1)
            queues[next_q].append(p)
        else:
            completed.append((p.name, time))
            
    return completed

# Example Run
procs = [Process("P1", 0, 8), Process("P2", 1, 4), Process("P3", 2, 10), Process("P4", 10, 2)]
print(mlfq_schedule(procs, [2, 4]))
```

## Edge Cases & Pitfalls

1.  **Starvation of Long Processes**:
    *   *Behavior*: If new short processes constantly arrive in Q0, processes in Q2 may never run.
    *   *Common Bug*: Implementing MLFQ without a "Priority Boost" mechanism (periodically moving all processes to Q0) causes system hang for background tasks.
2.  **I/O Bound Processes vs. CPU Bound**:
    *   *Behavior*: A process that gives up the CPU for I/O before its quantum expires should ideally stay in the same queue or move up.
    *   *Common Bug*: Demoting a process that yielded voluntarily. This penalizes I/O bound tasks (like UI responsiveness) by treating them as CPU-intensive.
3.  **Quantum Selection**:
    *   *Behavior*: If $TQ$ in Q0 is too large, MLFQ behaves like FCFS. If too small, context switching overhead dominates.
    *   *Common Bug*: Setting identical quanta for all levels, which defeats the purpose of the feedback mechanism.

# Technical breakdown: Named Pipes (FIFO)

## 1. Surgical Definition (Internals)
A **Named Pipe** is a persistent, kernel-resident FIFO (First-In, First-Out) object that facilitates Inter-Process Communication (IPC). Unlike anonymous pipes, which are transient and restricted to parent-child process hierarchies, named pipes exist in the **virtual filesystem** (on Windows: `\.\pipe`). The kernel treats it as a shared memory buffer managed via file-like semantics. It provides a reliable byte-stream or message-oriented channel between a server process and one or more client processes.

## 2. The Laboratory (Proof of Concept)
On Windows (Win32 API), the server creates the pipe using `CreateNamedPipe`.

```cpp
#include <windows.h>
#include <iostream>

int main() {
    // 1. Create the pipe handle
    HANDLE hPipe = CreateNamedPipe(
        "\.\pipe\KybernetesPipe", // Pipe name
        PIPE_ACCESS_DUPLEX,           // Read/Write access
        PIPE_TYPE_MESSAGE | PIPE_READMODE_MESSAGE | PIPE_WAIT, 
        1, 1024, 1024, 0, NULL);

    if (hPipe == INVALID_HANDLE_VALUE) return 1;

    // 2. Wait for a client to connect
    std::cout << "Waiting for client..." << std::endl;
    BOOL connected = ConnectNamedPipe(hPipe, NULL);
    
    if (connected) {
        char buffer[1024] = "Kybernetes System Online";
        DWORD written;
        WriteFile(hPipe, buffer, strlen(buffer), &written, NULL);
    }

    CloseHandle(hPipe);
    return 0;
}
```

## 3. Memory & System Context
* **Stack/Heap:** The `HANDLE` is a small stack-allocated pointer/index, but the actual buffer and pipe state live in the **Kernel Paged Pool**.
* **Cost:** Low CPU overhead, but kernel memory is consumed for the buffers (`nOutBufferSize` + `nInBufferSize`). Context switching occurs during read/write if the buffer is empty/full.
* **Life-cycle:** Persists as long as at least one handle to the pipe object exists in any process. Once all handles are closed, the kernel destroys the object.

## 4. Parameter Analysis (CreateNamedPipe)
1. **lpName:** The unique identifier. Must follow `\.\pipe\PipeName`. Necessary for clients to locate the resource.
2. **dwOpenMode:** Defines data flow (DUPLEX, INBOUND, OUTBOUND). Also enables `FILE_FLAG_OVERLAPPED` for asynchronous I/O.
3. **dwPipeMode:** Determines if data is read as a raw byte stream or discrete messages (`PIPE_TYPE_MESSAGE`).
4. **nMaxInstances:** Limits how many simultaneous client connections are allowed (1 to `PIPE_UNLIMITED_INSTANCES`). Prevents resource exhaustion.
5. **nOut/InBufferSize:** Advisory kernel buffer sizes. Passed to optimize memory usage based on expected payload size.
6. **nDefaultTimeOut:** Sets the default wait time for the `WaitNamedPipe` function if a client cannot immediately connect.
7. **lpSecurityAttributes:** Defines the Access Control List (ACL). Crucial for preventing unauthorized processes from hijacking the pipe.

## 5. Best Practices & Anti-Patterns
* **Idiomatic:** Use `PIPE_TYPE_MESSAGE` for structured data to avoid manual byte-framing.
* **Anti-Pattern:** Hardcoding pipe names without unique prefixes, which can lead to "Pipe Squatting" security vulnerabilities. Always use strict ACLs in production.

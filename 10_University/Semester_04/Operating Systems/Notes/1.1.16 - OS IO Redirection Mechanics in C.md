# Technical breakdown: I/O Redirection in C

## 1. Surgical Definition (Internals)
I/O redirection at the system level is the manipulation of the **File Descriptor Table (FDT)** within a process's Kernel-space structure. Each process starts with three standard streams: `stdin` (0), `stdout` (1), and `stderr` (2). Redirection involves remapping these specific indices to point to different entries in the **System-wide Open File Table**.

*   **The Lowest-Available Rule:** The Unix kernel guarantees that `open()` will always return the lowest available integer for a new file descriptor. This behavior is the "magic" that enables the Close/Open method.

## 2. The Laboratory (Proof of Concept)

### Method A: The Close/Open (Naive Redirection)
By closing a standard stream, its slot becomes the lowest available. The subsequent `open()` call claims it.

```c
#include <stdio.h>
#include <unistd.h>
#include <fcntl.h>

int main() {
    close(1); // Close STDOUT
    // open() will now return 1 because it's the lowest available
    int fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    
    printf("This goes into output.txt, not the terminal!\n");
    return 0;
}
```

### Method B: Open/Close/Dup/Close (Traditional `dup`)
This method is more explicit and allows for restoring the original stream if the descriptor is "saved" first.

```c
#include <unistd.h>
#include <fcntl.h>

int main() {
    int file_fd = open("log.txt", O_WRONLY | O_CREAT, 0644);
    close(1);         // Clear the slot
    dup(file_fd);     // Duplicates file_fd into slot 1
    close(file_fd);   // Close the extra reference
    
    write(1, "Redirection complete.\n", 22);
    return 0;
}
```
*Note: Modern systems prefer `dup2(file_fd, 1)`, which performs the close/dup atomically.*

## 3. Memory & System Context
*   **Stack/Heap:** Redirection logic lives in kernel-space tables; user-space pointers like `FILE *stdout` remain valid but their underlying file descriptor index now points elsewhere.
*   **Cost:** Negligible. It is a simple pointer update in the process control block. No data is copied; only the "mapping" changes.
*   **Life-cycle:** Redirections persist across `exec()` calls but are reverted when the process terminates, as the FDT is destroyed.

## 4. Best Practices & Anti-Patterns
*   **Edge Case (Race Conditions):** In multi-threaded programs, a thread might open a file between your `close(1)` and `open()`, stealing the FD 1 slot. Always use `dup2()` or `fcntl(F_DUPFD_CLOEXEC)` for thread safety.
*   **The "Silent Failure" Anti-Pattern:** Always check if `open()` returns -1. If you close `stdout` and `open()` fails, your program will have no standard output slot, leading to crashes in subsequent `printf` calls.
*   **Resource Leak:** Failing to close the original `file_fd` after `dup()` results in a leaked descriptor, which can hit the `RLIMIT_NOFILE` cap in long-running processes.

## Unix V CPU-Resident State Transitions: The Running and Ready Cycle
> **Seed:** "Analyze the internal state transitions of a memory-resident Unix V process, focusing on the movement between 'Created', 'User Running', 'Kernel Running', 'Ready to Run', and 'Preempted'. Explain the mechanical triggers for mode switching (traps, system calls, hardware interrupts), how the process context is preserved in the PCB and kernel stack, and the kernel's logic for re-scheduling a 'Preempted' process vs. one that voluntarily yielded."

## The Unix System V Process State Machine
A Unix System V process is an execution instance defined by its address space and its kernel-resident control structures. The transition between states is governed by the kernel's interrupt handlers and the `swtch()` (context switch) routine. Unlike modern multi-threaded models, the System V process model is strictly hierarchical, where every process (except process 0) is created via the `fork()` system call.

### Internal State Definitions
1.  **Created:** The process is in the transition of being initialized. The kernel has allocated a slot in the process table and a `u-area`, but the process is not yet eligible for execution.
2.  **Ready to Run (in Memory):** The process is loaded in main memory and resides in the run queue. It awaits the scheduler's selection.
3.  **User Running:** The CPU is executing instructions in the process’s virtual address space. The hardware is in "User Mode," restricting access to privileged instructions and kernel memory.
4.  **Kernel Running:** The CPU is executing kernel code on behalf of the process. This occurs after a mode switch. The process uses the kernel stack, not the user stack.
5.  **Preempted:** The process was executing in User Mode but was forcibly interrupted by the kernel (usually via a clock interrupt) to allow a higher-priority process or a fair-share time slice to execute.

## Mechanical Triggers for Mode Switching
Mode switching is the transition from **User Running** to **Kernel Running**. It is not a context switch; the process remains the "current" process, but its privilege level changes.

### 1. Traps (Synchronous)
Traps are internal events triggered by the execution of a specific instruction. 
*   **System Calls:** The process executes a special instruction (e.g., `INT 0x80` on x86 or `SVC` on ARM) to request a kernel service (e.g., `read()`, `fork()`). This is a "programmed exception."
*   **Faults:** Unexpected events like a division by zero or a page fault. The hardware forces a jump to a kernel-defined trap handler.

### 2. Interrupts (Asynchronous)
Interrupts are external hardware signals.
*   **Hardware Interrupts:** A disk controller signals I/O completion, or a keyboard sends a scan code.
*   **Clock Interrupts:** The programmable interval timer fires, signaling the end of a time quantum. This is the primary trigger for moving a process from **User Running** to **Preempted**.

## Context Preservation: PCB and Kernel Stack
When a transition occurs, the kernel must freeze the process's state to ensure it can resume later without data corruption.

### The Process Control Block (PCB)
The PCB (part of the `proc` structure in System V) resides in the kernel's permanent memory. It stores "global" process data:
-   **PID and PPID:** Identification.
-   **State:** The current state (e.g., `SRUN`, `SSLEEP`).
-   **Priority:** Used by the scheduler.
-   **Signal Bitmask:** Pending signals.

### The User Area (u-area) and Kernel Stack
The `u-area` contains data relevant only when the process is active. Within the `u-area` is the **Kernel Stack**.
-   **Mode Switch:** When a trap or interrupt occurs, the hardware automatically pushes the User Program Counter (PC) and Stack Pointer (SP) onto the **Kernel Stack**.
-   **Execution:** The kernel then uses this stack for local variables during the execution of the system call.
-   **Context Switch:** If the kernel decides to switch processes, it saves the remaining hardware registers (General Purpose Registers, Floating Point) onto the Kernel Stack and saves the Kernel SP in the PCB.

## Scheduling Logic: Preemption vs. Voluntary Yield
The kernel distinguishes between a process that *cannot* continue and one that was *forced* to stop.

### Voluntary Yield (The `sleep()` Mechanism)
When a process requests a resource (e.g., waiting for disk I/O), it calls `sleep()`. 
1.  The state changes from **Kernel Running** to **Sleep (Blocked)**.
2.  The kernel calls `swtch()` to select a new process.
3.  The process remains off the run queue until the resource becomes available (wakeup).

### Forced Preemption
When a clock interrupt occurs while a process is in **User Running** mode:
1.  The interrupt handler executes in **Kernel Running** mode.
2.  At the end of the interrupt handler, the kernel checks the `runrun` flag (indicating a higher priority process is ready or the quantum has expired).
3.  If `runrun` is set, the kernel changes the process state to **Preempted** (which is functionally a subset of **Ready to Run**).
4.  The process is placed back on the run queue. Unlike a yielding process, a preempted process is immediately eligible to run again if its priority is still the highest.

| Trigger | Origin | Final State | Mechanism |
| :--- | :--- | :--- | :--- |
| `read()` syscall | Internal | Sleep | Voluntary transition via `sleep()` |
| Page Fault | Internal | Kernel Running | Trap handler resolution |
| Clock Tick | External | Preempted | Interrupt handler + `swtch()` |
| `exit()` | Internal | Zombie | Final cleanup and deallocation |

## Failure Modes and Edge Cases
-   **Interrupting the Kernel:** In standard System V, the kernel is non-preemptible. If an interrupt occurs while in **Kernel Running** mode, the interrupt is handled, but the process returns to **Kernel Running** immediately. It cannot be moved to **Preempted** until it attempts to return to **User Mode**. This prevents race conditions on kernel data structures but increases scheduling latency.
-   **Kernel Stack Overflow:** Since the kernel stack is fixed-size within the `u-area`, deep recursion in kernel functions can lead to a system crash (Panic). This is why kernel-level algorithms must be iterative or shallow.

## Unix V Resource Management and Terminal States: Sleep, Swapping, and Zombie Dynamics

Unix System V process management is a finite state machine governed by kernel-mode transitions, where process context is preserved in the `proc` structure (always resident) and the `u_area` (swappable). The lifecycle is defined by the availability of two primary resources: the CPU and physical RAM.

## 1. The Taxonomy of Waiting: Asleep and Swapped States

In Unix V, a process enters a wait state when it requires a resource currently unavailable or awaits an external event.

### Asleep (Interruptible vs. Uninterruptible)
The kernel distinguishes between sleep depths based on the priority assigned during the `sleep()` call.
*   **Interruptible Sleep:** Occurs when the process waits for slow resources (e.g., terminal input). If a signal is sent to the process, the kernel wakes it immediately to handle the signal before the event occurs.
*   **Uninterruptible Sleep (Disk Sleep):** Occurs during critical I/O operations (e.g., waiting for a disk block). The kernel guarantees that the process will not be interrupted by signals until the I/O completes, preventing inconsistent metadata states. Signals are queued and processed only after the transition back to 'Ready to Run'.

### Swapped Out (Ready/Sleep)
When physical memory is exhausted, the **Swapper (Process 0)** identifies processes to move to the swap device.
*   **Ready to Run, Swapped:** The process has all resources to run except memory. It sits in the swap device queue.
*   **Asleep, Swapped:** The process is both waiting for an event and has been evacuated from memory. When the event occurs, it transitions to "Ready to Run, Swapped," awaiting the swapper to bring it back into RAM.

### Zombie State
A process enters the **Zombie (Exited)** state after executing `exit()`. The kernel releases all its memory and file descriptors, but the `proc` table entry remains. This entry stores the exit code and resource usage statistics, acting as a "death certificate" that persists until the parent process acknowledges it via `wait()`.

## 2. The Sleep/Wakeup Mechanism: Wait Channels
The Unix V `sleep`/`wakeup` mechanism is a non-preemptive synchronization primitive based on **Wait Channels**.

1.  **Sleep:** When a process calls `sleep(channel, priority)`, it provides a kernel address (the "channel") representing the event. The kernel sets the process state to `SSLEEP`, stores the channel in the `proc` structure, and calls `swtch()` to relinquish the CPU.
2.  **Wakeup:** When an event occurs (e.g., disk interrupt), the kernel calls `wakeup(channel)`. This is a broadcast mechanism: the kernel iterates through the `proc` table, and *every* process sleeping on that specific address is set to `SRUN` (Ready to Run).

**Analogy: The Paging System.** Think of `sleep` as a technician pressing a "Notify Me" button on a specific circuit breaker. `wakeup` is the alarm that triggers for everyone watching that breaker. They all rush to the scene, but only one might successfully grab the resource; the others may immediately `sleep` again (the "thundering herd" problem).

## 3. Lifecycle Termination: Exit/Wait Synchronization
Process termination is a two-stage handshake:

*   **exit(status):** The kernel disables signals, closes open files, and deallocates memory. It sets the state to `SZOMB`. It then sends a `SIGCHLD` signal to the parent.
*   **wait():** The parent process parses the `proc` table for any child in the `SZOMB` state. It collects the exit status and finally frees the `proc` table slot, effectively "reaping" the zombie.

**Failure Mode (Orphans):** If a parent dies before the child, the kernel re-parents the child to **Init (Process 1)**. Init is hardcoded to continuously call `wait()`, ensuring that no zombies remain indefinitely.

## 4. The Swapper (Process 0)
Process 0 is a system process created at boot. It does not execute user code. It operates on a 1-second heartbeat:
1.  **Swap Out:** It looks for processes in 'Asleep' state (prioritizing those asleep longest) or 'Ready' state (if memory is critically low).
2.  **Swap In:** It looks for processes in 'Ready Swapped' state and brings the one that has been on disk the longest into memory, provided there is space.

## 6. Technical Walkthrough and Edge Cases

### Transition: Kernel Running -> Preempted
While a process is in **Kernel Running**, it is non-preemptible. If an interrupt occurs and a higher-priority process becomes ready, the current process is not kicked out immediately. Instead, when it prepares to return to **User Running**, the kernel checks the `runrun` flag. If set, the process is moved to **Preempted** (effectively the same as Ready to Run) and the scheduler runs.

### Edge Case: Signal Handling in Sleep
If a process is in **Interruptible Sleep** and receives a `SIGKILL`:
1.  The kernel changes its state from `SSLEEP` to `SRUN`.
2.  The process is placed on the run queue.
3.  When it eventually executes, the `sleep()` call returns an error (usually `EINTR`).
4.  The kernel's trap handler sees the pending signal and forces the process into the `exit()` logic before it can execute another user instruction.

### Edge Case: The "Unkillable" Process
A process in **Uninterruptible Sleep** (waiting for a faulty NFS mount or a stuck hardware controller) will ignore `SIGKILL` (kill -9). Since the kernel never checks signals while in the middle of a critical disk-wait `sleep()`, the process stays in the `proc` table until the hardware responds. If the hardware never responds, the process is "stuck" until reboot.

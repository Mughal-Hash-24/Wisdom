# Technical breakdown: Read & Write File Descriptors

## 1. Surgical Definition (Internals)
In UNIX-like systems, a **File Descriptor (FD)** is a non-negative integer that acts as a per-process unique identifier for an open I/O stream. The kernel maintains a **Process Open File Table** where the FD serves as an index. Each entry points to a system-wide **File Table Entry**, which contains the file offset, status flags (O_APPEND, etc.), and a reference to the **vnode/inode**.

### Syntax Construction:
*   `ssize_t read(int fd, void *buf, size_t count);`
*   `ssize_t write(int fd, const void *buf, size_t count);`

**Case Analysis:**
1.  **Success (`> 0`):** Returns the number of bytes actually processed. For `read`, this may be less than `count` if EOF is near or if reading from a pipe/socket.
2.  **EOF (`0` during `read`):** Indicates the end of the file or the closure of the writing end of a pipe.
3.  **Error (`-1`):** The operation failed. The global `errno` is set.
    *   `EBADF`: `fd` is not a valid or open file descriptor.
    *   `EINTR`: The call was interrupted by a signal before any data was read/written.
    *   `EAGAIN/EWOULDBLOCK`: Non-blocking I/O is set, and no data is currently available/buffer is full.

## 2. The Laboratory (Proof of Concept)
This C++ snippet demonstrates manual handling of a partial `write` to ensure the entire buffer is committed to the kernel.

```cpp
#include <unistd.h>
#include <iostream>
#include <cstring>
#include <errno.h>

void safe_write(int fd, const char* msg) {
    size_t remaining = strlen(msg);
    size_t total_written = 0;
    while (remaining > 0) {
        ssize_t n = write(fd, msg + total_written, remaining);
        if (n == -1) {
            if (errno == EINTR) continue; // Interrupted by signal, retry
            perror("write failed");
            return;
        }
        total_written += n;
        remaining -= n;
    }
}

int main() {
    const char* data = "Kybernetes Kernel: Writing to STDOUT_FILENO
";
    safe_write(1, data); // 1 is STDOUT
    return 0;
}
```

## 3. Memory & System Context
*   **Stack/Heap:** The `buf` typically resides in the process's Stack or Heap (User-space). 
*   **Cost:** Each `read`/`write` triggers a **Context Switch**. The CPU moves from Ring 3 (User) to Ring 0 (Kernel). The Kernel copies data from the User buffer to the Kernel's Page Cache (or vice versa).
*   **Life-cycle:** FDs are allocated via `open()` or `pipe()` and must be explicitly released via `close()`. Leaking FDs can exhaust the per-process limit (`RLIMIT_NOFILE`).

## 4. Best Practices & Anti-Patterns
*   **Best Practice:** Always wrap `read`/`write` in loops to handle partial transfers and `EINTR`.
*   **Anti-Pattern:** Assuming `write(fd, buf, count)` always writes `count` bytes. In network sockets or pipes, it often writes less if the kernel buffer is full.
*   **Anti-Pattern:** Using FDs without checking for `-1` return values, leading to silent failures or memory corruption.

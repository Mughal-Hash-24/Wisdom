# Three level Scheduling
> **Seed:** "@expand Explain in detail the concept of three level scheduling with examples and diagrams"

Three-level scheduling is a hierarchical process management architecture that optimizes system throughput, response time, and resource utilization by managing process state transitions across different storage tiers and temporal resolutions. It decomposes the scheduling problem into three distinct layers: Long-term (Admission), Medium-term (Memory), and Short-term (CPU) scheduling.

## The Hierarchical Architecture

The system operates as a tiered pipeline where processes are filtered and throttled as they move closer to the CPU.

### 1. Long-Term Scheduler (Admission Scheduler)
The Long-term Scheduler (LTS) determines which programs are admitted to the system for processing. It selects processes from a "Job Pool" (typically on disk) and loads them into memory into the "Ready Queue."

*   **Mechanism:** Its primary objective is to maintain a stable **Degree of Multiprogramming** (the number of processes in memory). It must balance the "Job Mix"—ensuring a healthy ratio of I/O-bound processes (which spend most time waiting for hardware) and CPU-bound processes (which saturate the processor).
*   **Temporal Resolution:** Executes infrequently (seconds or minutes).
*   **Failure Mode:** If the LTS admits too many processes, the system overhead for management exceeds productive work. If it admits only CPU-bound jobs, I/O devices sit idle; if only I/O-bound, the CPU remains underutilized.

### 2. Medium-Term Scheduler (Memory/Swapping Scheduler)
The Medium-term Scheduler (MTS) manages the residency of processes in physical RAM. It is the core component of **swapping** logic.

*   **Mechanism:** When memory pressure increases or a process becomes blocked for an extended duration, the MTS removes the process from main memory and stores it on the "Swap Space" (disk). This is known as *swapping out*. When resources become available or the process is ready to run again, the MTS *swaps it in*.
*   **Temporal Resolution:** Executes at an intermediate frequency (hundreds of milliseconds to seconds).
*   **Failure Mode:** **Thrashing**. If the MTS swaps processes in and out too aggressively, the system spends more time performing disk I/O for swap operations than executing instructions.

### 3. Short-Term Scheduler (CPU Scheduler/Dispatcher)
The Short-term Scheduler (STS) selects which process from the Ready Queue (in-memory) will execute next on the CPU.

*   **Mechanism:** It is invoked by events such as timer interrupts (preemption), I/O requests, or system calls. It uses algorithms (Round Robin, Shortest Job First, Multi-level Feedback Queues) to decide the next process and hands control to the **Dispatcher** to perform the context switch.
*   **Temporal Resolution:** Executes extremely frequently (milliseconds).
*   **Failure Mode:** **Starvation**. If the algorithm is poorly tuned (e.g., strict priority), low-priority processes may never receive CPU time.

```text
       |
       | (Long-term Scheduler) Admittance
       v
[ READY QUEUE (RAM) ] <--------+
       |                       |
       | (Short-term)          | (Medium-term)
       | Dispatch              | Swap-in
       v                       |
[     CPU     ]                |
       |                       |
       +----(Block/Wait)-------+--> [ SUSPENDED QUEUE (Disk) ]
       |                            ^
       +----(Medium-term)-----------+
             Swap-out
```

## Mechanical Analogy: The Logistics Factory
To visualize the internals, consider a high-security manufacturing plant:

1.  **The Gatekeeper (LTS):** Thousands of trucks (jobs) wait outside the factory gates. The Gatekeeper looks at the factory floor and decides which trucks are allowed to enter the loading docks based on how much space is left.
2.  **The Floor Manager (MTS):** Once inside, if a truck is waiting for a part that hasn't arrived yet, the Floor Manager orders it to move to a "Holding Lot" (Swap Space) outside to clear the internal dock for a truck that is ready to work.
3.  **The Crane Operator (STS):** On the actual assembly line, the Crane Operator moves from one active truck to the next every 10 seconds, performing a small piece of work on each to keep all lines moving simultaneously.

## Comparative Dynamics

| Feature | Long-Term (LTS) | Medium-Term (MTS) | Short-Term (STS) |
| :--- | :--- | :--- | :--- |
| **Speed** | Slowest | Intermediate | Fastest |
| **Location** | Disk to RAM | RAM to Disk (Swap) | RAM to CPU |
| **Core Goal** | Job Mix/Degree of MP | Memory Optimization | CPU Utilization/Latency |
| **Trigger** | Process Creation | Memory Pressure | Interrupt/Clock Tick |

## Critical Edge Cases: The Trashing Threshold
A critical failure occurs when the interaction between the MTS and STS breaks down. If the STS detects low CPU utilization (because many processes are blocked waiting for I/O), it may signal the LTS to admit more jobs. However, if the low utilization is actually caused by memory pressure (processes waiting for pages to be swapped in), admitting more jobs increases memory demand. This creates a positive feedback loop where the system spends 99.9% of its cycles on the MTS/LTS swapping logic and 0.1% on the STS executing code. A robust kernel must implement a **Working Set Model** to prevent the LTS from admitting new jobs when the page fault rate exceeds a specific threshold.

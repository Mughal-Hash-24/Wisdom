# Deep Dive: Standard Streams and File Descriptors (0, 1, 2)

## 1. Ontological Definition
In Unix-like operating systems, **File Descriptors (FDs)** are non-negative integers that serve as abstract handles for accessing file-like objects (files, sockets, pipes, devices). By convention, every new process is born with three specific FDs pre-allocated: **0 (stdin)**, **1 (stdout)**, and **2 (stderr)**. These are the "Standard Streams," providing a unified interface for a process to communicate with its environment without knowing the specific hardware or file system details.

## 2. The Internal Mechanics (Under the Hood)
When a process is created, the Kernel maintains a **File Descriptor Table** in the process's `task_struct`. This table is an array of pointers to `file` objects in the system-wide open file table.

*   **Logic:** When you call `read(0, buf, len)`, the kernel looks at index 0 of the process's FD table, finds the associated file object (usually the terminal's input buffer), and pulls data.
*   **Redirection (`dup2`):** The magic of `>` or `|` happens via the `dup2(oldfd, newfd)` system call. If you run `ls > out.txt`, the shell forks `ls`, opens `out.txt` as FD 3, and calls `dup2(3, 1)`. This overwrites index 1 in the FD table with the pointer from index 3. Now, `ls` "thinks" it is writing to stdout (1), but the kernel sends the data to `out.txt`.
*   **Math/Pseudo-code:**
    ```cpp
    // Conceptual Kernel FD Table
    struct Process {
        File* fd_table[MAX_FDS];
    };
    // Redirection: stdout (1) = open("out.txt")
    process->fd_table[1] = kernel_open("/path/to/out.txt");
    ```

## 3. Systems Context & Anchoring (The Workshop Analogy)
Think of a process as a **Workstation**. 
*   **FD 0 (stdin)** is the **In-Tray**: Material comes in here.
*   **FD 1 (stdout)** is the **Out-Tray**: Finished products go here.
*   **FD 2 (stderr)** is the **Trash/Alarm**: Scraps or warning lights go here.

In **C++**, `std::cin`, `std::cout`, and `std::cerr` are object wrappers around these FDs. `std::cout << "Hello"` eventually resolves to a `write(1, "Hello", 5)` syscall. This abstraction allows the same code to print to a screen, a file, or a network socket simply by changing what the FD points to.

## 4. Edge Cases & Constraints
*   **Buffering Disparity:** `stdout` is often line-buffered (sent when `
` is hit) when connected to a terminal, but block-buffered when redirected to a file. `stderr` is usually **unbuffered** to ensure error messages appear immediately even if the system crashes.
*   **FD Leaks:** If a process opens files but never closes them, it can hit the `ulimit` (max FDs), causing new connections or file opens to fail.
*   **Closing 0, 1, 2:** If a process closes FD 1 and then opens a new file, the kernel will assign FD 1 (the lowest available) to that new file. Subsequent `printf` calls will then "leak" into that file, which is a common source of bugs in low-level daemon processes.

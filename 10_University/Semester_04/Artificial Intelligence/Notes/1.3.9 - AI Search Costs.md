# Deep Dive: Local Cost in Search Space

## 1. Ontological Definition
**Local Cost** (also known as *Step Cost* or *Edge Weight*) is the immediate expenditure required to transition from a current state $s$ to a successor state $s'$ via an action $a$. It is an atomic unit of cost that does not account for the history of the path or the remaining distance to the goal.

## 2. The Internal Mechanics (Under the Hood)
### Mathematical Representation
In a graph $G = (V, E)$, the local cost is defined as a function $c$:
$$c(s, a, s') 	o \mathbb{R}_{\geq 0}$$
Where:
- $s$: Current State
- $a$: Action taken
- $s'$: Successor State

In most implementations, this is simplified to $w(e)$ where $e = (s, s')$.

### Python Implementation
```python
class Graph:
    def __init__(self):
        # Adjacency list: {state: {neighbor: local_cost}}
        self.edges = {
            'A': {'B': 5, 'C': 2},
            'B': {'D': 1},
            'C': {'B': 1, 'D': 8}
        }

    def get_local_cost(self, current, action, successor):
        """Returns the atomic step cost"""
        return self.edges.get(current, {}).get(successor, float('inf'))

# Usage
g = Graph()
print(f"Cost A -> B: {g.get_local_cost('A', 'move', 'B')}") # Output: 5
```

## 3. Systems Context & C++ Anchor
In low-level engineering (C++), local costs are often stored in an **Adjacency Matrix** (for dense graphs) as a 2D array `double weights[N][N]` or an **Adjacency List** (for sparse graphs) using `std::vector<std::pair<int, double>>`.
- **Memory Impact:** A matrix costs $O(V^2)$ space, while an list costs $O(V + E)$.
- **Cache Locality:** Matrices provide better cache performance for dense lookups but waste memory on null edges.

# Deep Dive: Global Cost (Path Cost)

## 1. Ontological Definition
**Global Cost** ($g(n)$) is the cumulative expenditure incurred to reach a specific node $n$ from the start state $s_{start}$. Unlike local cost, it represents the "historical debt" of the search agent.

## 2. The Internal Mechanics (Under the Hood)
### Mathematical Representation
The global cost of a path $P = (s_0, s_1, ..., s_k)$ is the sum of all local costs along that path:
$$g(s_k) = \sum_{i=1}^{k} c(s_{i-1}, a_i, s_i)$$
Recursive definition used in algorithms like UCS and A*:
$$g(s') = g(s) + c(s, a, s')$$

### Python Implementation (Path Tracker)
```python
def calculate_global_cost(path, graph):
    total_cost = 0
    for i in range(len(path) - 1):
        step_cost = graph.get_local_cost(path[i], 'move', path[i+1])
        total_cost += step_cost
    return total_cost

# Example Path
path = ['A', 'C', 'B', 'D']
# Costs: (A->C: 2) + (C->B: 1) + (B->D: 1) = 4
```

## 3. Systems Context & C++ Anchor
In pathfinding algorithms, $g(n)$ is usually stored in a **Distance Map** (e.g., `std::unordered_map<Node, double> dists`).
- **Update Logic:** During edge relaxation, we check `if (dists[u] + weight < dists[v])`. This is the core of Bellman-Ford and Dijkstra.
- **Data Type:** Must use `double` or `float` for continuous weights, or `uint64_t` for large integer weights to prevent overflow.

## 4. Edge Cases & Constraints
- **Negative Costs:** Global cost calculation fails with algorithms like Dijkstra if local costs are negative (leads to infinite loops or wrong paths). Requires Bellman-Ford.
- **Zero Costs:** Can cause infinite cycles in uninformed searches if cycles exist.
